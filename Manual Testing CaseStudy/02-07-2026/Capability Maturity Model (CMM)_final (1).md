# Capability Maturity Model (CMM)

# 

*The Capability Maturity Model (CMM) is a framework developed by the Software Engineering Institute to measure and improve the maturity of an organization's software development processes.*

*CMM is a process improvement model that helps organizations improve the quality, efficiency, and maturity of their software development processes through five maturity levels.*





* ## Objectives of CMM
1. Improve software quality.
2. Reduce project risks.
3. Deliver projects on time and within budget.
4. Increase customer satisfaction.
5. Continuously improve development processes.



### 

* ### The 5 Levels of CMM



#### 

#### Level 1 – Initial (Ad Hoc)

**Definition:** At this level, the software development process is unorganized and unpredictable. Success depends mainly on individual effort.

**Characteristics:**

1. No standard process.
2. Poor planning.
3. Frequent delays.
4. High risk.
5. Difficult to repeat successful projects.

**Example:** A startup develops software without any documented process. Every developer works differently.



#### Level 2 – Repeatable (Managed)

**Definition:** Basic project management processes are established so similar projects can be repeated successfully.

**Characteristics:**

1. Project planning is done.
2. Requirements are managed.
3. Cost and schedule are tracked.
4. Basic quality assurance exists.
5. Previous project experience is reused.

**Example:** A company follows a standard process for every mobile app project, making similar projects easier to manage.



#### Level 3 – Defined

**Definition:** The organization has well-documented and standardized software development processes that everyone follows.

**Characteristics:**

1. Standard processes across the organization.
2. Proper documentation.
3. Employee training.
4. Defined roles and responsibilities.
5. Better communication.

**Example:** Every team follows the same SDLC, coding standards, testing process, and documentation guidelines.

#### 

#### Level 4 – Quantitatively Managed

**Definition:** The organization measures and controls software processes using metrics and statistical techniques.

**Characteristics:**

1. Performance is measured.
2. Quality metrics are collected.
3. Defects are analyzed.
4. Decisions are data-driven.
5. Predictable project outcomes.

**Example:** The company tracks defect density, testing coverage, productivity, and delivery time to improve performance.

#### 

#### Level 5 – Optimizing

**Definition:** The organization continuously improves its processes using innovation and feedback.

**Characteristics:**

1. Continuous process improvement.
2. Root cause analysis.
3. New technologies are adopted.
4. Automation is encouraged.
5. Focus on preventing defects rather than fixing them.

**Example:** A company regularly improves its development process by introducing AI-based testing tools, CI/CD pipelines, and automation based on project feedback.
